Disciple: Design System

I took charge of building and maintaining the design system for Disciple, a SaaS product that lets community leaders create their own independent community platforms on web, iOS and Android.

Background

Disciple already had an asset library in use, but it was poorly organised and lacked any documentation. This made it difficult to use and understand, leading to components being inconsistently designed and implemented.

Solution

By establishing a true design system, a clear set of standards would be provided. These would guide design decisions and coordinate the creation, organisation and implementation of components.

Outcome

The design process saw a significant increase in efficiency. Likewise, the UI benefitted from greater consistency and is on track to be improved visually, now that the asset library is well organised.

Examination

We started by examining the designs and systems Disciple already had in place. This was to identify any repeating design patterns that could be implemented into the new design system, with or without modification. This gave us a foundation to build upon, rather than from the ground up.

Colour

Disciple’s brand colours were implemented into the design system as a set of colour styles. Because they were due to be changed as part of a large scale rebrand, they were given less descriptive names like “primary”. This way there would be no need to rename any styles or classes to prevent confusion.

The product presented a different challenge. This was because customers are able to customise their community platform's colour scheme. Luckily, there was already a system in place, where each UI element is assigned a class that controls its values. These values are then set by the customer in their CMS.

We created a set of distinct colour styles to correspond with these classes. While these are not exactly “pretty” to look at in designs, they are easily distinguishable and therefore easy to identify, minimising any room for error.

These styles are split into categories based on where they should be used. For example, “wall” colours must only be used on the background of the community platform. Using a colour style on a background style from a different category was forbidden, as doing so always led to issues, such as poor colour contrast.

Typography

We chose Proxima Nova as our typeface because of its eligibility and geometric appearance pairing well with Disciple’s SaaS product. Rather than purchasing a license for every font available, we limited our options to light, regular, bold and italic. This was to keep costs down most of all, but had the added benefit of encouraging a clear cut visual hierarchy by limiting options to 4 fonts rather than 48.

With the typeface chosen we added a number of font styles to the design system. These were used to create two type scales. We decided on two type scales because the typography requirements of Disciple’s marketing platforms and the product differed slightly.

Layout

We found that the spacing and sizing of elements was often decided on a case by case basis. These were two of the main inconsistencies throughout both the product and marketing website. Our solution to this was a scale that would determine the size of elements and spacing between them. This would bring spatial harmony to the product and website, allowing them to be perceived and higher in quality and more trustworthy.

I introduced this as an 8 point scale. By using an even number as a base, elements could be aligned, without worrying about decimal units complicating measurements. Additionally, it provided a good balance between flexibility and a manageable number of variables. A 4 point scale was also introduced for elements that required more intricate layouts inside of components.

Finally, I also made sure to outline how interactive elements should be laid out to accommodate for sufficient target sizes. This ensured that all interactive elements would be easily accessible to the customer and end-users.

Iconography

Throughout existing designs, outlined and filled icons from different sources were being used with no real logic behind them. We therefore decided to overhaul iconography. The Ionic icon pack was used as a base for a quick solution without the need to design an entire library ourselves. Outlined icons would be used as the “default” states, while their filled versions would show an active state where necessary.

When creating a custom icon, designers would be required to do so at a size of 16x16 pixels with a stroke width of 1. The stroke would then need to be expanded, to ensure that it scales correctly after being exported as an SVG. Once we had enough custom icons, we would create our own icon pack for designers and developers to use.

Asset library

While taking inventory on all of the components in the asset library, several steps were taken to improve its organisation. The first was to identify any components that shared the same function and decide which one should be used while removing the other(s). We also removed any components that were too specific to be reused and were therefore redundant. The next step was influenced by our decision to organise the asset library following the atomic design framework. Because there were several larger “organism” components that had repeated elements of their own, we extracted those smaller elements and made them “molecule” and “atom” components.

The main advantage of organising components in this way is that it provided a clear method for locating components, increasing the design team’s efficiency. Atomic design also encourages designers to see interfaces as both a collection of parts and the whole they combine to create. This helps them make the best design decisions for each individual component and the final UI simultaneously.
